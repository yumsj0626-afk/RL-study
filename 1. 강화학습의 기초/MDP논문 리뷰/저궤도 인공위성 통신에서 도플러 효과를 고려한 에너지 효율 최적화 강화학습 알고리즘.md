
# 논문 리뷰: 저궤도 인공위성 통신에서 도플러 효과를 고려한 에너지 효율 최적화 강화학습 알고리즘

**Kim & Hong, 경희대학교 | 2023 한국컴퓨터종합학술대회**

---

## 읽게 된 계기

모듈 3 과제에서 직접 MDP를 정의하는 과제가 있었는데, 어떻게 하면 MDP를 좀 더 잘 설계할 수 있을지 궁금해서 인용률 높은 논문을 찾아봤다.

읽으면서 느낀 건, MDP 자체는 오래된 프레임워크인데 도메인만 바뀌어도 완전히 새로운 문제가 된다는 거였다. 모델 자체의 활용성은 무궁무진하다는 걸 다시 한번 느꼈다.

---

## 한 줄 요약

LEO 위성 간 통신에서 도플러 효과로 인한 채널 변화를 MDP로 모델링하고, PPO 기반 강화학습으로 에너지 효율을 최적화한 연구.

---

## 문제 정의

위성 통신에는 두 가지 핵심 문제가 동시에 발생한다.

- **도플러 효과**: 위성이 빠르게 움직이기 때문에 채널 상태가 수시로 변한다.
- **에너지 효율**: 위성의 수명 자체와 직결되는 문제다.

기존 연구들은 2D 격자 형태로 토폴로지를 단순화했는데, 이 논문은 실제 **원형 궤도** 환경을 그대로 모델로 삼았다는 게 차별점이다.

또 위성 네트워크는 토폴로지가 수시로 바뀌는 애드혹(Ad-hoc) 네트워크라서, 수식 기반의 라우팅 테이블로는 유연하게 대응하기 어렵다. 그래서 강화학습을 쓴 거고, 이 선택 자체가 납득이 됐다.

---

## MDP 설계 분석

이 논문에서 제일 집중해서 본 부분이다.

### State

$$S = \{LOS, V, \theta, d\}$$

| 변수 | 의미 |
| :--- | :--- |
| $LOS$ | Non-LOS=0, 현재 위성=1, 목적지=2, LOS=3 |
| $V$ | 도플러 효과로 측정한 상대속도 리스트 |
| $\theta$ | LOS 범위 내 위성들의 전송각 리스트 |
| $d$ | 상대속도 기반으로 측정한 거리 리스트 |

도메인의 물리량을 그대로 state로 가져온 게 핵심이라고 느꼈다.

도플러 공식에서 나오는 변수들을 state에 직접 반영했고, LOS 여부를 범주형으로 인코딩해서 "지금 통신이 가능한가"를 상태 안에 녹인 설계가 깔끔했다.

**물리 현상 → 수식 → state 변수** 로 이어지는 흐름이 일관적이었다.

### Action

$$A = \{I_{i+1}\}$$

다음 홉 위성의 인덱스를 선택하는 것. 라우팅 문제니까 "어디로 보낼지"만 결정하면 되고, action space를 단순하게 유지한 게 오히려 학습 안정성에 도움이 됐을 것 같다.

### Reward

$$R = \beta(\gamma)^H \sum_{i=0}^{H-1} \frac{C_i}{P_{t,i}}$$

에너지 효율(채널 용량 ÷ 소비 전력)을 보상으로 직접 정의했다.

목적함수와 보상 함수가 1:1로 대응되는 구조가 인상적이었다. 홉 수에 따른 감쇠 상수 $\gamma$를 곱해서 경로가 길어질수록 보상이 줄어드는 설계도 자연스럽게 경로 효율을 유도한다.

## 실험 결과

궤도당 위성 수를 늘리면서 성능을 비교했다.

| 궤도당 위성 수 | 평균 누적 보상 |
| :--- | :--- |
| 8개 | 8.103 |
| 12개 | 32.801 |
| 16개 | 114.335 |

위성이 많을수록 LOS 범위 내 선택지가 늘어나고, 에너지 효율이 좋은 경로를 찾을 확률이 높아지기 때문에 보상이 급격히 증가한 것으로 해석된다.

PPO를 선택한 것도 납득이 된다. 위성 환경처럼 상태가 연속적으로 변하는 환경에서는 정책 업데이트가 너무 크게 튀면 학습이 불안정해지는데, PPO의 clipping이 그걸 잡아주기 때문이다.

## 내가 MDP 설계할 때 가져갈 것

이 논문을 읽고 나서 정리한 MDP 설계 원칙이다.

**1. 도메인의 핵심 물리량을 state에 직접 녹여라.**

"이 상황에서 agent가 좋은 결정을 내리려면 뭘 알아야 하나?" 를 먼저 생각하고, 그 답이 곧 state가 돼야 한다. 이 논문은 도플러 공식에서 나오는 변수들을 그대로 state로 가져왔고, 그게 설계를 깔끔하게 만들었다.

**2. reward는 목적함수와 직접 연결시켜라.**

최적화하고 싶은 것을 reward로 정의하면 설계가 단순해지고, 나중에 결과를 해석하기도 훨씬 쉬워진다.

**3. action space는 최대한 단순하게 유지해라.**

결정해야 할 것만 action으로 정의하고, 나머지는 environment에 맡기는 게 낫다.
