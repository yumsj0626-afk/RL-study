
# Dynamic Programming 기반 최적 정책 탐색 보고서

## 1. 과제 목적과 문제를 바라본 방식

이번 과제는 ParkingWorld 환경에서 Dynamic Programming을 이용해 주차 가격 정책을 평가하고, 더 나은 정책을 찾아가는 과정을 구현한 것이다. 환경의 상태는 현재 몇 개의 주차 공간이 사용 중인지를 의미하고, 행동은 주차 가격을 얼마로 설정할지를 의미한다. 낮은 가격은 주차 수요를 늘리는 방향으로 작용하고, 높은 가격은 수요를 줄이는 방향으로 작용한다. 문제의 핵심은 단순히 “주차장이 많이 차면 좋다”가 아니라, “많이 차는 것은 좋지만 완전 만차는 피하고 싶다”는 조건이 들어가 있다는 점이다.

처음에는 이 과제가 단순히 Bellman update 공식을 코드로 옮기는 문제처럼 보였다. 하지만 실행 결과를 플롯으로 확인하면서, 이 과제의 핵심은 공식 암기가 아니라 “가치 함수가 정책을 어떻게 평가하고, 그 평가가 다시 정책 개선으로 어떻게 이어지는가”를 확인하는 데 있다는 것을 알 수 있었다. 내가 정리했던 이론 흐름으로 보면, 이번 과제는 Policy Evaluation, Policy Improvement, Policy Iteration, Value Iteration, 그리고 GPI가 실제 코드에서 어떻게 연결되는지 확인하는 작업이었다.

이번 보고서에서는 내가 구현한 세 개의 핵심 함수와 실행 결과를 중심으로 정리한다. 특히 단순히 정답 코드가 통과했는지가 아니라, 결과 플롯이 무엇을 의미하는지, 왜 state 8과 state 9에서 다른 행동이 선택되는지, 그리고 Policy Iteration과 Value Iteration이 왜 같은 최적 정책에 도달하는지를 중심으로 이론을 점검한다.

## 2. Policy Evaluation: 현재 정책이 만들어내는 가치 확인

첫 번째로 구현한 함수는 `bellman_update`이다. 이 함수는 특정 상태 `s`에 대해 현재 정책 `pi`를 그대로 따른다고 했을 때의 가치 `V[s]`를 갱신한다. 여기서 중요한 점은 이 단계가 최적 행동을 고르는 단계가 아니라는 것이다. 즉, 가능한 행동 중 가장 큰 값을 선택하는 것이 아니라, 현재 정책이 각 행동을 선택할 확률에 따라 평균을 내야 한다.

```python
new_v = 0.0

for a in env.A:
    action_prob = pi[s, a]
    action_value = 0.0
    for sp, (r, p) in enumerate(env.transitions(s, a)):
        action_value += p * (r + gamma * V[sp])
    new_v += action_prob * action_value

V[s] = new_v
```

이 코드는 Bellman expectation update를 구현한 것이다. 각 행동에 대해 가능한 다음 상태의 보상과 다음 상태 가치를 이용해 `q(s,a)`를 계산하고, 그 값을 현재 정책의 행동 확률로 가중 평균한다. 즉, 이 함수는 “이 상태에서 현재 정책대로 행동한다면 앞으로 얼마나 좋은 결과를 기대할 수 있는가?”를 계산한다.

실행에서는 모든 상태에서 가격 1을 선택하는 고정 정책을 평가했다. 결과 가치 함수는 다음과 같은 흐름을 보였다.

| State | Value |
| ----: | ----: |
|     0 | 80.04 |
|     1 | 81.65 |
|     2 | 83.37 |
|     3 | 85.12 |
|     4 | 86.87 |
|     5 | 88.55 |
|     6 | 90.14 |
|     7 | 91.58 |
|     8 | 92.81 |
|     9 | 93.78 |
|    10 | 87.77 |

처음 이 플롯을 봤을 때 중요한 해석 포인트는, 가치 함수가 state 9까지는 증가하다가 state 10에서 떨어진다는 점이었다. 이는 문제의 보상 구조와 일치한다. 주차 공간이 많이 사용될수록 사회적 효용은 증가하지만, 완전 만차는 비상 수요를 수용할 수 없기 때문에 좋지 않은 상태로 평가된다. 그래서 state 10은 단순히 “가장 많이 찬 상태”임에도 value가 낮게 나온다.

오른쪽 policy 플롯에서는 모든 상태에서 action 1만 선택하는 검은 줄이 나타났다. 이 정책은 상태가 낮든 높든 같은 가격을 유지하는 단순한 baseline 정책이다. 여기서 내가 확인한 점은, Policy Evaluation은 정책을 바꾸는 과정이 아니라 현재 정책이 만들어내는 세계를 관찰하는 과정이라는 것이다. 즉, 가치 함수는 “현재 정책이 이 환경에서 장기적으로 어떤 결과를 만드는지”를 보여주는 평가표에 가깝다.

## 3. Policy Iteration: 평가 결과를 이용해 정책을 바꾸기

두 번째 구현은 `q_greedify_policy`이다. 이 함수는 현재 가치 함수 `V`를 기준으로 각 행동의 q값을 계산하고, 가장 좋은 행동만 선택하도록 정책을 바꾼다. 여기서부터는 Policy Evaluation이 아니라 Policy Improvement이다. 따라서 현재 정책의 행동 확률 `pi(a|s)`를 곱하지 않는다. 모든 행동을 후보로 놓고, 가장 큰 q값을 가진 행동을 선택한다.

```python
q_values = []

for a in env.A:
    q = 0.0
    for sp, (r, p) in enumerate(env.transitions(s, a)):
        q += p * (r + gamma * V[sp])
    q_values.append(q)

best_action = np.argmax(q_values)

pi[s] = np.zeros(len(env.A))
pi[s, best_action] = 1
```

이 코드에서 가장 중요한 부분은 `np.argmax(q_values)`이다. 이 한 줄이 현재 가치 함수 기준으로 greedy한 행동을 고르는 부분이다. 내가 정리했던 이론의 표현으로 말하면, `V`를 통해 현재 정책의 결과를 평가한 뒤, 그 평가 결과를 이용해 더 좋은 정책으로 이동하는 과정이다. 즉, Policy Improvement Theorem이 코드로 나타난 부분이라고 볼 수 있다.

Policy Iteration을 실행한 결과 최종 정책은 다음과 같았다.

| State | Value | Action |
| ----: | ----: | -----: |
|     0 |  81.6 |      0 |
|     1 |  83.3 |      0 |
|     2 |  85.0 |      0 |
|     3 |  86.8 |      0 |
|     4 |  88.5 |      0 |
|     5 |  90.2 |      0 |
|     6 |  91.7 |      0 |
|     7 |  93.1 |      0 |
|     8 |  94.3 |      0 |
|     9 |  95.3 |      3 |
|    10 |  89.5 |      3 |

이 결과는 처음 고정 정책보다 훨씬 해석 가능한 정책이다. state 0부터 state 8까지는 action 0을 선택하고, state 9와 state 10에서는 action 3을 선택한다. 이를 주차 가격 문제로 해석하면, 주차 공간이 아직 여유 있을 때는 낮은 가격을 설정해서 점유율을 높이고, 거의 만차에 가까워지면 높은 가격을 설정해서 더 이상 점유율이 올라가지 않도록 막는 정책이다.

대화 중에 가장 깊게 짚었던 부분은 state 8과 state 9의 차이였다. 왜 state 8에서는 낮은 가격(action 0)이 최적인데, state 9에서는 높은 가격(action 3)이 최적일까? 이 차이는 즉시 보상만 봐서는 잘 보이지 않는다. Bellman 관점에서는 현재 보상뿐 아니라 다음 상태의 가치까지 같이 본다. state 8은 아직 만차까지 여유가 있으므로 낮은 가격을 통해 state 9처럼 더 가치 높은 상태로 갈 가능성을 키우는 것이 좋다. 반면 state 9는 이미 가장 좋은 상태에 가까우므로, 낮은 가격을 유지하면 state 10으로 넘어갈 위험이 커진다. state 10은 가치가 낮기 때문에, state 9에서는 높은 가격을 설정해 만차 위험을 줄이는 것이 더 좋은 선택이 된다.

이 지점에서 내가 정리했던 “좋은 정책은 좋은 상태에 오래 머무르게 하고, 나쁜 상태로 가는 전이를 줄인다”는 말이 실제 결과로 확인되었다. state 8의 action 0은 더 좋은 상태로 이동할 기회를 키우는 행동이고, state 9의 action 3은 나쁜 상태로 넘어갈 위험을 줄이는 행동이다. 둘은 서로 다른 행동처럼 보이지만, 실제로는 같은 원리에서 나온 결정이다. 즉, 둘 다 장기 기대 수익을 높이는 방향이다.

## 4. Value Iteration: 정책을 직접 들고 가지 않아도 최적 정책이 나오는 이유

세 번째 구현은 `bellman_optimality_update`이다. 이 함수는 Value Iteration에서 사용된다. 앞의 Policy Iteration은 평가와 개선을 분리해서 수행했다. 반면 Value Iteration은 가치 함수 업데이트 안에 이미 max 연산이 들어가 있다. 그래서 정책을 명시적으로 계속 들고 가지 않아도 된다.

```python
q_values = []

for a in env.A:
    q = 0.0
    for sp, (r, p) in enumerate(env.transitions(s, a)):
        q += p * (r + gamma * V[sp])
    q_values.append(q)

V[s] = max(q_values)
```

이 코드에서 핵심은 `V[s] = max(q_values)`이다. Policy Evaluation에서는 정책 확률로 평균을 냈고, Policy Improvement에서는 q값이 가장 큰 행동을 골라 정책을 바꿨다. Value Iteration에서는 이 둘 중 improvement에 해당하는 max 선택이 가치 업데이트 안으로 들어온다. 그래서 Value Iteration은 정책을 따로 평가하고 개선하는 구조라기보다, 매번 최적 행동을 가정하며 가치 함수를 갱신하는 구조로 볼 수 있다.

실행 결과 Value Iteration도 Policy Iteration과 같은 정책을 얻었다. state 0부터 state 8까지는 action 0, state 9와 state 10에서는 action 3을 선택했다. 가치 함수의 형태도 거의 같았다. 이 결과는 중요한 의미가 있다. 알고리즘의 절차는 다르지만, 둘 다 결국 Bellman optimality equation을 만족하는 최적점으로 수렴한다는 것이다.

대화 중에 확인했던 질문은 “왜 Value Iteration에서는 pi를 안 써도 최적 정책이 나오느냐?”였다. 답은 max 연산에 있다. `max_a`가 이미 현재 가치 함수 기준으로 가장 좋은 행동을 선택하는 역할을 하기 때문에, 명시적인 정책이 없어도 업데이트 과정 안에 greedy improvement가 포함되어 있다. 이후 가치 함수가 수렴하면, 마지막에 그 가치 함수에 대해 greedy policy를 복원하면 된다.

또한 `value_iteration2`는 Policy Iteration과 Value Iteration의 관계를 더 잘 보여주었다. 이 방식은 각 상태마다 먼저 `q_greedify_policy`로 local하게 정책을 greedy하게 만들고, 곧바로 `bellman_update`로 값을 갱신한다. 즉, 전체 정책을 완전히 평가한 뒤 개선하는 Policy Iteration과 달리, 상태별로 작은 평가와 작은 개선을 섞어 수행한다. 이게 내가 정리했던 GPI 관점과 연결된다.

## 5. 결과 기반 이론 점검과 내 이해의 변화

이번 실행 결과를 통해 가장 확실하게 정리된 것은, Bellman update가 단순한 수식 계산이 아니라 의사결정의 기준이라는 점이다. `r + gamma * V[sp]`는 지금 당장의 보상과 다음 상태의 가치를 함께 본다. 그래서 state 8과 state 9처럼 겉으로 비슷해 보이는 상태에서도 다른 행동이 선택된다. 강화학습에서 중요한 것은 현재 상태만 보는 것이 아니라, 이 행동이 다음 상태 분포를 어떻게 바꾸고 그 결과 장기 가치가 어떻게 달라지는지를 보는 것이다.

또 하나의 인사이트는 Value Function과 Policy를 분리해서 봐야 한다는 점이다. Value Function은 각 상태가 장기적으로 얼마나 좋은지를 보여준다. Policy는 그 가치를 높이기 위해 어떤 행동을 선택하는지를 보여준다. 처음 고정 가격 정책의 플롯은 “현재 정책이 만들어낸 상태 가치”를 보여주었고, Policy Iteration 이후의 플롯은 “그 가치를 바탕으로 개선된 행동 전략”을 보여주었다. 이 둘을 함께 보니, 정책이 단순히 정답 테이블이 아니라 가치 평가를 바탕으로 만들어진 결과라는 점이 더 분명해졌다.

Policy Iteration과 Value Iteration의 차이도 더 명확해졌다. Policy Iteration은 평가와 개선이 분리되어 있어 학습 구조를 이해하기 쉽다. 먼저 현재 정책을 충분히 평가하고, 그다음 greedy하게 개선한다. 반면 Value Iteration은 평가와 개선이 한 업데이트 안에 압축되어 있다. 처음에는 두 알고리즘이 꽤 다른 방식처럼 보였지만, 실행 결과 같은 최적 정책으로 수렴하는 것을 보면서 둘이 같은 목표를 향한 다른 구현 방식이라는 점을 확인했다.

GPI 관점에서는 이번 과제 전체를 하나의 흐름으로 볼 수 있었다. Policy Evaluation은 현재 정책의 가치를 맞추는 방향이고, Policy Improvement는 현재 가치 함수에 대해 정책을 greedy하게 만드는 방향이다. 이 두 과정이 서로 밀고 당기면서 최적점으로 이동한다. Policy Iteration은 이 과정을 크게 나누어 수행하고, Value Iteration은 더 촘촘하게 섞어 수행한다. 결국 중요한 것은 평가와 개선이 반복적으로 상호작용한다는 점이다.

## 6. 결론

이번 과제는 내가 정리했던 동적 프로그래밍 이론을 코드와 실행 결과로 점검한 작업이었다. `bellman_update`를 통해 Policy Evaluation이 현재 정책의 장기 가치를 계산하는 과정임을 확인했고, `q_greedify_policy`를 통해 그 가치 함수를 기반으로 정책이 개선되는 과정을 확인했다. 또한 `bellman_optimality_update`를 통해 Value Iteration이 정책을 명시적으로 유지하지 않아도 max 연산을 통해 최적 행동 선택을 업데이트 안에 포함한다는 점을 이해했다.

실행 결과는 이론과 잘 맞았다. 고정 정책에서는 모든 상태에서 같은 가격을 선택했지만, 최적 정책은 상태에 따라 가격을 다르게 설정했다. 낮은 점유 상태에서는 낮은 가격으로 점유율을 높이고, 높은 점유 상태에서는 높은 가격으로 만차 위험을 줄였다. 특히 state 8과 state 9의 행동 차이는 Bellman 관점의 의사결정이 무엇인지 잘 보여주는 사례였다.

결국 이번 과제의 핵심은 테스트를 통과한 코드 자체보다, 코드와 플롯을 통해 DP의 의사결정 구조를 확인한 데 있다. 내가 이해한 흐름으로 정리하면, Policy Evaluation은 현재 정책을 이해하는 과정이고, Policy Improvement는 그 이해를 바탕으로 더 나은 선택을 하는 과정이다. Value Iteration은 이 두 과정을 더 압축적으로 섞은 형태이며, 이 전체 구조는 GPI라는 큰 틀 안에서 이해할 수 있다. 앞으로 TD Learning이나 Q-Learning을 배울 때도, 이번 과제에서 확인한 “평가와 개선의 반복”을 기준으로 보면 모델 없는 강화학습 알고리즘들이 무엇을 근사하는지 더 명확하게 이해할 수 있
